1. Contexto del mini proyecto
La empresa ficticia Ventas Express S.A.S. registra sus ventas en una hoja de calculo. El archivo
funciona para capturar datos, pero presenta redundancia, errores de actualizacion, dificultades para
consultar informacion y riesgo de inconsistencias. El equipo de tecnologia requiere convertir esa
estructura en una base de datos relacional normalizada para PostgreSQL.
Problema central:


Una misma venta puede incluir varios productos, pero en la tabla original los productos estan
agrupados en campos separados por comas.



Los datos de clientes, vendedores, categorias, productos y medios de pago se repiten en cada
registro.



El total de la venta se guarda como dato capturado, aunque puede calcularse desde el detalle.



No existen claves primarias, claves foraneas ni restricciones de integridad.



El modelo actual dificulta responder consultas basicas como ventas por vendedor, productos mas
vendidos o historial de compras por cliente.

2. Objetivos de aprendizaje


Identificar problemas de redundancia, dependencia parcial, dependencia transitiva y atributos
no atomicos.



Aplicar Primera Forma Normal (1FN), Segunda Forma Normal (2FN) y Tercera Forma Normal
(3FN).



Definir entidades, atributos, claves primarias, claves foraneas y relaciones cardinales.



Diseñar un modelo entidad-relacion para un sistema de ventas.



Implementar el modelo fisico en PostgreSQL usando DDL, restricciones y datos de prueba.



Validar que el modelo normalizado permite reconstruir la informacion original sin perdida de
datos.

3. Reglas de negocio del sistema de ventas
1. Un cliente puede realizar muchas ventas; cada venta pertenece a un unico cliente.
2. Un vendedor puede atender muchas ventas; cada venta es atendida por un unico vendedor.
3. Una venta puede contener uno o varios productos.
4. Un producto puede aparecer en muchas ventas.
5. Cada producto pertenece a una categoria.
6. El precio unitario usado en una venta debe conservarse en el detalle, porque el precio del
producto puede cambiar con el tiempo.
7. El descuento se registra por linea de detalle, no por producto general.
8. El medio de pago se registra para cada venta.
9. El total de la venta debe poder calcularse desde el detalle: cantidad * precio_unitario - descuento.
10. No debe permitirse registrar un detalle de venta con producto inexistente, venta inexistente o
cantidad menor o igual a cero.

6. Actividades paso a paso
Paso 1. Diagnostico de la tabla cruda


Identifica atributos no atomicos.



Marca datos repetidos por cliente, vendedor, producto, categoria y pago.



Explica al menos tres anomalias: insercion, actualizacion y eliminacion.



Redacta una lista inicial de dependencias funcionales.

Paso 2. Aplicacion de 1FN


Transforma los campos multivaluados en registros separados.



Cada fila debe representar una venta con un solo producto asociado.



Verifica que no existan listas separadas por comas ni grupos repetitivos.



Propone una clave compuesta provisional para la tabla en 1FN.

Paso 3. Aplicacion de 2FN


Evalua si la tabla en 1FN usa una clave compuesta.



Separa los atributos que dependen solo de una parte de la clave.



Identifica entidades como venta, cliente, vendedor, producto y detalle de venta.



Define claves primarias y foraneas para las entidades resultantes.

Paso 4. Aplicacion de 3FN


Elimina dependencias transitivas.



Separa categorias de productos si la categoria depende del producto y no de la venta.



Separa ciudades o zonas solo si justificas que se administraran como catalogos.



Revisa que cada atributo dependa de la clave, de toda la clave y de nada mas que la clave.

Paso 5. Diseno del modelo entidad-relacion


Construye el diagrama ER con entidades, atributos, claves y cardinalidades.



Incluye como minimo: clientes, vendedores, ventas, detalle_venta, productos y categorias.



Puedes agregar metodos_pago o ciudades si tu justificacion lo requiere.



No uses la tabla ventas_crudas como tabla transaccional final.

Paso 6. Implementacion fisica en PostgreSQL


Crea el esquema normalizado con CREATE TABLE.



Paso 7. Validacion con consultas SQL



9. Consultas minimas de validacion
El modelo final debe permitir responder estas preguntas con SQL. No basta con dibujar el modelo.
-- 1. Total calculado por cada venta desde el detalle.
-- Resultado esperado: venta_id, total_calculado.
-- 2. Productos mas vendidos por cantidad total.
-- Resultado esperado: producto_codigo, producto_nombre, unidades_vendidas.
-- 3. Ventas por vendedor.
-- Resultado esperado: vendedor_id, vendedor_nombre, cantidad_ventas, valor_total.
-- 4. Historial de compras de un cliente especifico.
-- Resultado esperado: cliente_doc, cliente_nombre, venta_id, fecha_venta, producto, cantidad.
-- 5. Control de integridad: detalles sin venta o sin producto.
-- Resultado esperado: cero registros huerfanos.

10. Restricciones tecnicas obligatorias


Usar PostgreSQL como motor de base de datos.



Crear un esquema propio, por ejemplo normalizacion_ventas.



No modificar manualmente los datos para esconder inconsistencias; las transformaciones deben
justificarse.



No dejar campos multivaluados en el modelo final.



No guardar total_venta como unica fuente de verdad. Si decides almacenarlo, debes justificar la
decision y validar su consistencia.



Usar claves foraneas para proteger la integridad referencial.



Usar CHECK para validar cantidad mayor que cero y precios no negativos.



Usar UNIQUE donde existan identificadores naturales como documento de cliente o codigo de
producto.

15. Pistas pedagogicas sin entregar la solucion


No empieces creando tablas al azar: primero escribe las dependencias funcionales.



Si un dato se repite muchas veces y describe a una entidad, probablemente debe vivir en su
propia tabla.



Si una tabla de detalle tiene una clave compuesta, revisa dependencias parciales para aplicar
2FN.



Si un atributo depende de otro atributo que no es clave, revisa dependencia transitiva para
aplicar 3FN.



El precio de un producto en una venta historica no siempre debe actualizarse cuando cambia el
precio actual del catalogo.



Un buen modelo permite reconstruir la venta original usando JOIN, pero sin guardar listas
separadas por comas.

16. Checklist final antes de entregar


[ ] La tabla final no contiene productos_codigos, cantidades o precios separados por comas.



[ ] Cada tabla tiene una clave primaria clara.



[ ] Las relaciones tienen claves foraneas activas.



[ ] Las cantidades no aceptan valores menores o iguales a cero.



[ ] Los productos no se duplican innecesariamente.



[ ] Los clientes no se duplican por cada compra.



[ ] El total de la venta se puede calcular con una consulta SQL.



[ ] El diagrama ER coincide con los scripts SQL entregados.



[ ] El README explica como ejecutar los scripts.



[ ] Las consultas de validacion se ejecutan sin errores.
